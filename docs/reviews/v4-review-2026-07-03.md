# v4 Tutorial Review — Accuracy, Consistency, Clarity

**Date:** 2026-07-03
**Scope:** `v4/README.md`, `v4/pre-work-setup.md`, `v4/workshop-build-deploy.md`, cross-checked against `prd/ecommerce-analytics.md`, `data/sales-data.csv`, the repo root `README.md`, and the actual repo structure.

## Summary

The v4 docs are in strong shape overall: internal navigation links all resolve, the two-part structure is coherent, terminology (milestones vs. plan steps, TASKS.md vs. Claude's in-session todo list) is used consistently, and most factual claims check out. The review found **one high-impact accuracy problem** (the sample dataset does not match the dollar figures the PRD and tutorial reference), **two false statements in the v4 text itself**, and a handful of smaller consistency and clarity issues, listed by priority below.

---

## 1. High priority — accuracy

### 1.1 The dataset doesn't match the PRD's "Expected Output" (students will think their dashboard is broken)

Computed directly from `data/sales-data.csv`:

| Metric | PRD claims | Actual data |
|---|---|---|
| Total Sales | ~$650,000 – $700,000 | **$116,500.21** |
| Total Orders | 482 | 482 ✓ |
| Record count ("Data Volume") | ~1,000 records | **482 records** |
| Top category | "Electronics or Audio (varies)" | Electronics ($42,684); **Audio is 3rd** (Wearables 2nd at $23,698, Audio $19,638) |
| Date range | 12 months | 2024-01-03 → 2024-12-31 ✓ |
| Categories / regions | 5 / 4 | 5 / 4 ✓ (all `total_amount` = `quantity × unit_price` ✓) |

The PRD is even internally inconsistent: its Data Volume section says ~1,000 records while its Expected Output table says 482 orders.

**Why this matters for v4 specifically:** the PRD's acceptance criterion "Data loads correctly: All values match expected calculations from CSV" is what students validate against, and v4 Section 1 sends them to the PRD's Acceptance Criteria. A student whose dashboard correctly shows ~$116,500 will compare it to the PRD's ~$650K and conclude their code is wrong.

**v4 also propagates the wrong figure:** the Streamlit intro example at `v4/workshop-build-deploy.md:382` uses `st.metric("Total Sales", "$650,000")`, reinforcing the PRD's number.

**Fix options:** (a) correct the PRD's Expected Output table and Data Volume section to match the shipped CSV (~$116,500, 482 records, Electronics > Wearables > Audio), and change the `st.metric` example to `"$116,500"`; or (b) regenerate `sales-data.csv` to match the PRD. Option (a) is a two-file text edit; option (b) invalidates the deployed reference dashboard's numbers.

### 1.2 Part 1 recap claims a step that never happened

`v4/pre-work-setup.md:844` — the "What's next" recap says Part 1 accomplished:

> "Initialized Claude Code so it understands your project (AI context)"

Part 1 contains no `/init` step. `/init` happens in Part 2, Section 4.4 — which explicitly explains why it's done at the *end* ("at the start there's nothing to describe"). This bullet directly contradicts that rationale and will make students wonder if they missed a setup step. **Fix:** delete the bullet. (Same line exists in v3's setup doc, so this looks like an inherited leftover.)

### 1.3 "You'll only ever type one slash command for the entire tutorial" is false

`v4/pre-work-setup.md:425`. The tutorial itself instructs students to type:

- `/exit` (Section 2.4 and throughout)
- `/mcp` (Section 2.6)
- `/config` (Part 2, Section 1, step 2)
- `/init` (Part 2, Section 4.4)
- plus `/logout` and `/plugin list` in troubleshooting

The intended point is that the *Superpowers skills* never require slash commands. **Fix:** rephrase to something like "Note that installing the plugin is the only time you'll type a slash command to *use Superpowers* — the skills themselves activate from plain-English prompts."

### 1.4 Windows Git installer option label is wrong

`v4/pre-work-setup.md:299` says to select **"Use Git from Git Bash and also from 3rd-party software"**. The Git for Windows installer's recommended option reads **"Git from the command line and also from 3rd-party software"** ("Use Git from Git Bash only" is a *different*, more restrictive option). As written, students will be looking for a radio button that doesn't exist — and the doc's own "Heads up: focus on the goal, not exact labels" only partially mitigates it because the two real options are easy to confuse. **Fix:** use the actual label.

## 2. Medium priority — consistency

### 2.1 Traceability example pairs TASK-1 with TASK-2's work

`v4/workshop-build-deploy.md:492` (Key Concept: Traceability):

> `TASK-1 in TASKS.md --> Commit "TASK-1: add KPI scorecards" --> ...`

In the sample board (Section 1), TASK-1 is "Project setup and data loading"; KPI scorecards is **TASK-2**. In the section that teaches traceability, the example itself doesn't trace correctly. **Fix:** change the commit message to `"TASK-1: set up project and data loading"` (which also matches the example commit message at line 505).

### 2.2 Final checklist says milestones came from the plan — they came from the PRD

`v4/workshop-build-deploy.md:751`: "Milestones created **from the plan** (4-8 of them...)". Section 1 creates milestones from the **PRD**, and Section 3's comparison table explicitly says milestones "Come from: the PRD, up front (Section 1)" while the plan comes later. **Fix:** "Milestones created from the PRD".

### 2.3 TDD example presents a made-up number as this dataset's value

`v4/workshop-build-deploy.md:269`: "You write a test that says 'compute_total_sales should return **$1,234,567 for this dataset**'". The actual dataset total is $116,500.21. The `$1,234,567` placeholder is fine in the README's ASCII mockup, but "for this dataset" turns it into a factual claim — and a student who later sees their test assert ~$116,500 may be confused. **Fix:** use `$116,500` (pairs well with fixing 1.1), or drop "for this dataset."

### 2.4 Stated total time doesn't match the section estimates

`v4/workshop-build-deploy.md:5` says "about 3 hours" (v4 README says "~3 hours"), but the section labels sum to ~100 minutes (15 + 30 + 5 + 35 + 15). Part 1's 70–100 min claim *does* match its section sum (25 + 50 + 15 + 10 = 100). If 3 hours reflects real-world pace (Q&A, watching builds, reading artifacts), consider a note saying the labels are focused-work time; otherwise reconcile the numbers.

### 2.5 v4 README says every task is built test-first; Part 2 says only flagged tasks are

`v4/README.md:79`: executing-plans described as "implementing **each task** test-first." Part 2 (Sections 2.2e and 4.1) repeatedly says only *flagged* tasks (data transformations) get TDD, and rendering tasks skip it. **Fix:** "...implementing each task, test-first where the plan flags it."

### 2.6 Section 4.3 cycle diagram orders steps differently than the instructions

`v4/workshop-build-deploy.md:544`: the diagram shows `"Let's work on TASK-N" → executing-plans works its plan steps → Move to In Progress` — i.e., the board is updated *after* implementation. The prompt pattern (4.1 and 4.3) says move to In Progress *first, then* implement. **Fix:** reorder the diagram boxes.

### 2.7 Repo folder listings are stale

`v4/pre-work-setup.md:587` and `:647` tell students to expect `data/`, `v1/`, `v2/`, and `prd/` in their clone. The repo now also contains `v3/`, `v4/`, and `docs/` — and notably, **v4 (the version they're following) isn't in the list**, which invites "am I in the right repo?" doubt. **Fix:** mention `v4/` (and ideally `docs/`) or say "folders like `data/`, `prd/`, and the versioned tutorial folders (`v1/`–`v4/`)."

### 2.8 `ls` in the Part 2 prerequisites check vs. the Windows guidance in Part 1

Part 1 (`pre-work-setup.md:220`) teaches "On Windows, use `dir` instead" of `ls`, but Part 2's prerequisites (`workshop-build-deploy.md:64`) have everyone run `ls data/sales-data.csv`. This works in PowerShell (where `ls` is an alias) but fails in cmd, and it silently contradicts Part 1's guidance. **Fix:** add a `# Windows (cmd): dir data\sales-data.csv` alternative or note that Cursor's default Windows terminal is PowerShell, where `ls` works.

### 2.9 Workshop TOC omits a section

The "Other Superpowers skills you'll meet later" section (`workshop-build-deploy.md:841`) sits between Troubleshooting and Glossary but isn't listed in the table of contents (`:9–26`). **Fix:** add it.

### 2.10 Glossary ordering slip

`workshop-build-deploy.md:880–881`: **Superpowers** is listed after **TASKS.md**, breaking the otherwise-alphabetical order. **Fix:** move Superpowers above TASKS.md.

## 3. Low priority — accuracy nits and claims to verify externally

### 3.1 macOS Python notes are slightly dated

- `pre-work-setup.md:354` and the troubleshooting entry (`:737`) suggest macOS's `python` may be "Python 2.x". macOS hasn't shipped Python 2 since 12.3 (2022); on a current Mac, `python` typically returns "command not found" rather than an old version. The `python3` guidance is still right — just soften/update the Python 2 framing.
- `pre-work-setup.md:340` repeats the "Python install manager — ignore it" note in the **macOS** steps; that option is a Windows-only offering on python.org. Harmless but potentially confusing ("I don't see that option"). Keep it in the Windows steps only.
